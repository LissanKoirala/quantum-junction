# Quantum Junction: QMill Challenge

This repository contains our Quantum Peak hackathon pipeline for finding the
hidden output bitstring of obfuscated peaked quantum circuits. The core idea was
not to brute-force every circuit. Instead, we built a method-selection layer that
inspects each `.qasm` file and routes it to the cheapest plausible solver:
statevector when exact simulation is still realistic, MPS sampling when the
entanglement structure is manageable, and structure-aware tensor-network methods
when the circuit looks deliberately obfuscated.

Our focus was to develop a time- and compute-efficient pipeline that could move
beyond brute force. The challenge rewards exact bitstrings, but it also rewards
originality, generalization, and computational efficiency. That shaped the
engineering: every expensive run had to be justified by static circuit evidence.

## Challenge

Each challenge circuit starts from `|0...0>` and produces one secret bitstring
with much higher probability than the other outputs. The task is to recover that
peak bitstring exactly.

The circuits were generated by hiding RX-encoded target patterns inside composed
blocks of the form `U U^dagger`, inserting additional RX-only layers, and then
applying an obfuscation engine. In practice, that means the best attack is not
always a larger simulator. The better question is:

> Which structure is still visible in the QASM, and which solver can exploit it
> before the computation becomes too expensive?

All candidate bitstrings in this repository use Qiskit/counts order: the
right-most bit corresponds to qubit `0`.

## Winning Strategy

We used multiple approaches because no single method was reliable across all
difficulty bands.

### 1. Exact Statevector Baseline

For small circuits, we used Qiskit statevector simulation. This gave exact peak
probabilities and exact answers for the calibrating instances. It also fixed the
bit-order convention for all later methods.

Statevector was intentionally limited to small circuits because memory and time
scale as `2^n`. Past roughly `28` qubits, it stopped being the right first move
for the hackathon setting.

### 2. Qiskit Aer MPS Sampling

For larger circuits where statevector was too costly, we used Qiskit Aer's
matrix product state simulator with shot-based sampling. The main pilot settings
used `4096` shots and bond dimensions in the `32-64` range, with additional
trials over multiple seeds and settings when a candidate needed stability
checks.

This solved several moderate cases and generated useful candidates, but it also
showed a clear failure mode: when the circuit required larger bond dimensions,
SVD truncation and low-bond compression could lose useful amplitude information.
Flat sampled distributions and unstable top-1 candidates were treated as warning
signals, not as submission-quality evidence.

### 3. Kremer-Dupuis Tensor-Network Methods

For harder circuits, we adapted the Kremer-Dupuis peaked-circuit methods:

- Low-bond MPS with bitstring distillation.
- Tensor Network Operator midpoint contraction.
- MPO iterative cancellation with unswapping.

These methods exploit the fact that the challenge circuits are not random
circuits. They contain hidden inverse blocks, repeated interactions, disguised
permutations, and cancellation opportunities. The important step was deciding
which structural attack to try first.

### 4. Graph/Tree-Ordered Tensor Networks

The hardest non-very-hard circuits showed long-range and repeated CX structure
that made plain one-dimensional MPS ordering fragile. We built graph/tree-aware
ordering around the interaction graph:

- Nodes are qubits.
- Edge weights come from repeated `cx` and `swap` interactions.
- Orderings include graph-derived CPU/GPU variants such as RCM, degree, fast,
  midpoint, and identity fallbacks.
- Candidates are sampled from graph-ordered Quimb `CircuitMPS` runs and then
  normalized into a shared evidence table.

This became the main source of approximate answers for easy, moderate, and some
hard circuits.

### 5. MPO Sweep for the Hardest Circuits

For the hardest circuits, we also implemented an MPO sweep. Instead of using one
fixed MPO insertion point, we inserted MPOs at fragile regions suggested by the
hidden inverse-block structure of the circuits. We combined this with
tree-inspired ordering and subspace reduction: when multiple sweeps agreed on
certain bit positions, those high-confidence bits were frozen and later searches
focused only on uncertain positions.

This ran for most of Saturday and required too much compute to finish within the
hackathon. The approach is still important because it goes beyond the exact
template in the challenge. It searches for useful MPO insertion sites and
structural signals, so the same idea can apply to other obfuscated circuits that
contain hidden cancellations, repeated motifs, inverse blocks, or non-random
interaction patterns.

## Quantum Peak Method Selector

The central engineering contribution is `quantum_peak_method_selector.py`. It is
a dependency-free QASM triage tool that inspects each circuit before we spend
simulation time on it.

It measures:

- Number of qubits and operations.
- Counts for `rx`, `rz`, `cx`, and `swap`.
- Approximate depth.
- Entangling-graph density.
- Repeated two-qubit pairs.
- Leading and trailing one-qubit dressing layers.
- Angle-grid clues, including rotations close to simple fractions of `pi`.
- Hidden-permutation and inverse-block signals.

It then recommends the first serious route:

- Exact statevector baseline.
- Low-bond MPS with bitstring distillation.
- Tensor Network Operator midpoint contraction.
- MPO iterative cancellation with unswapping.

This saved substantial computational time because we avoided blindly running
every method on every circuit. The selected route was passed into the
tensor-network pipeline, which then used low-bond MPS sampling, midpoint TNO
contraction, or central-MPO cancellation with graph/tree-based qubit ordering
and unswapping.

## Usage

Analyze one circuit:

```bash
python3 quantum_peak_method_selector.py examples/tiny_peak.qasm
```

Analyze all challenge files and write a markdown report:

```bash
python3 quantum_peak_method_selector.py challenges --out reports/qmill_method_report.md --detail-limit -1
```

Write machine-readable reports:

```bash
python3 quantum_peak_method_selector.py challenges --format json --out reports/qmill_method_report.json
python3 quantum_peak_method_selector.py challenges --format csv --out reports/qmill_method_report.csv
```

Run one Aer MPS trial from a generated config:

```bash
python3 agent_work/mps_distill/scripts/run_mps_trial.py \
  --config-file agent_work/mps_distill/configs/pilot_easy_hard.jsonl \
  --task-id 1 \
  --out-dir agent_work/mps_distill/results
```

Run one graph/tree-ordered Quimb candidate search:

```bash
python3 jobs/quimb_tree_tensor_runner.py \
  --qasm challenges/easy/challenge-40_16.qasm \
  --out-dir outputs/tree_tensor_sim/quimb_pilot \
  --backend auto \
  --max-bond 512 \
  --samples 2048
```

Run one MPO-unswapping attempt:

```bash
python3 jobs/peaked_mpo_unswap_runner.py \
  --qasm challenges/hard/challenge-64_40.qasm \
  --out-dir outputs/tree_tensor_sim/peaked_unswap_gpu \
  --backend auto \
  --max-bond 8192 \
  --samples 1000
```

## Limitations

Approximate tensor-network candidates are not automatically correct. We treated
them differently depending on the evidence:

- Exact statevector rows are ground truth for the small instances.
- Stable repeated MPS/Quimb winners are stronger than one-off samples.
- Flat distributions, all-unique samples, and unstable top candidates are weak
  evidence.
- Marginal-threshold strings can be wrong when lossy compression shifts a bit
  near the `0.5` threshold.
- The very-hard circuits need more targeted structure exploitation or more
  compute than was available during the hackathon.
